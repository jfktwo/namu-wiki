# 01. 구조잡기

## 문제 정의

조경 업체가 입찰을 따면 특정 수종·규격·수량의 나무가 필요하지만, 근처에 재고를 가진 공급자를 찾을 방법이 없다.
공급자는 개인 홈페이지 등 각자의 매체에 재고를 올리지만 수요자에게 닿지 않는다.
**수요자와 공급자가 만나는 장(場)이 없다.**

---

## 서비스 개념

**"나무 장터"** — 수목 수요 ↔ 공급 매칭 플랫폼

```
수요자 (조경업체)              공급자 (묘목장/개인)
"왕벚나무 3년생 50그루    →    "내 재고: 왕벚나무 3년생
 필요합니다"                    80그루, 충남 천안"
                    ← 매칭 알림
```

---

## 핵심 기능

### 1. 수요 게시판 (조경업체가 등록)
- 수종 / 규격(수령) / 수량 / 필요 시기 / 지역 / 예산

**예시**
| 수종 | 규격 | 수량 | 시기 | 지역 |
|------|------|------|------|------|
| 왕벚나무 | 3년생 | 50그루 | 2026-07 | 경기도 |
| 왕벚나무 | 10년생 | 50그루 | 2026-07 | 경기도 |
| 소나무 | 5년생 | 50그루 | 2026-07 | 경기도 |
| 담쟁이 | 1년생 | 10그루 | 2026-07 | 경기도 |

### 2. 재고 등록 (공급자가 등록)
- 수종 / 규격 / 수량 / 단가 / 위치 / 사진 / 유효기간(자동 만료)

### 3. 매칭 알림
- 수요 등록 시 → 근처 공급자에게 푸시 알림
- 재고 등록 시 → 매칭되는 수요자에게 알림

## 기술 스택

| 레이어 | 선택 | 이유 |
|--------|------|------|
| 앱 | React Native (Expo) | iOS/Android 동시 지원, AI 코드 생성 친화적 |
| 백엔드 | Supabase | DB + 인증 + 실시간 + 파일 저장 올인원 |
| 지도 | Kakao Map API | 한국 주소/지역 처리 최적 |
| 알림 | Supabase Realtime | 별도 서버 없이 실시간 알림 가능 |

## DB 구조 (초안)

### users
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | 기본키 |
| role | enum | buyer(조경업체) / seller(공급자) |
| name | text | 업체명 |
| phone | text | 연락처 |
| region | text | 주요 활동 지역 |

### demands (수요)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | 기본키 |
| user_id | uuid | 수요자 |
| species | text | 수종 |
| age_year | int | 수령(년) |
| quantity | int | 수량 |
| region | text | 납품 지역 |
| budget | int | 예산(원) |
| deadline | date | 필요 시기 |
| status | enum | open / matched / closed |

### stocks (재고)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | 기본키 |
| user_id | uuid | 공급자 |
| species | text | 수종 |
| age_year | int | 수령(년) |
| quantity | int | 수량 |
| price | int | 단가(원) |
| lat / lng | float | 위치 좌표 |
| photos | text[] | 사진 URL 목록 |
| expires_at | date | 재고 유효기간 |
