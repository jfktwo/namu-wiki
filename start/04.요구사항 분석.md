# 04. 요구사항 분석

나무 장터 MVP의 1차 요구사항을 정리한다.

이번 단계는 **간단 회원가입/로그인 + 수요/재고 CRUD + 단순 매칭**까지를 범위로 한다.

---

## 1. 목표

수목 수요자와 공급자가 각각 수요와 재고를 등록하고, 조건에 맞는 상대 게시글을 매칭 후보로 확인할 수 있게 한다.

MVP에서는 거래, 채팅, 결제까지 만들지 않고, 매칭 후보를 확인한 뒤 문의 요청을 남기는 흐름까지만 구현한다.

---

## 2. 포함 범위

- 간단 회원가입/로그인
- 가입 시 역할 고정: `buyer`, `seller`, `admin`
- 수요 등록, 목록, 상세, 수정, 삭제
- 재고 등록, 목록, 상세, 수정, 삭제
- 좌표 기반 50km 이내 단순 매칭
- 가격 조건 기반 매칭
- 수요자와 공급자 양쪽 매칭 결과 조회
- 매칭 후보에 대한 문의 요청
- 관리자 게시글 숨김 처리

---

## 3. 제외 범위

- 채팅
- 결제
- 배송/운송 관리
- 실시간 푸시 알림
- 복잡한 매칭 점수 알고리즘
- 사용자 정지
- 사용자 역할 변경
- 관리자 직접 수정/삭제

---

## 4. 사용자 역할

### 수요자

조경업체, 시공사처럼 특정 수종, 규격, 수량의 나무가 필요한 사용자다.

수요자는 수요 글을 등록하고, 본인 수요에 맞는 재고 매칭 후보를 확인한다.

### 공급자

묘목장, 농장, 개인 판매자처럼 판매 가능한 재고를 가진 사용자다.

공급자는 재고 글을 등록하고, 본인 재고에 맞는 수요 매칭 후보를 확인한다.

### 관리자

서비스 운영자다.

관리자는 수요/재고 게시글을 확인하고, 문제가 있는 게시글을 숨김 처리한다.

---

## 5. 인증 요구사항

### 회원가입

사용자는 이메일과 비밀번호로 가입한다.

필수 입력값:

| 항목 | 설명 |
|------|------|
| 이메일 | 로그인 ID |
| 비밀번호 | 로그인 비밀번호 |
| 역할 | `buyer`, `seller`, `admin` 중 하나 |
| 이름/업체명 | 사용자 또는 업체 표시명 |
| 전화번호 | 문의용 연락처 |
| 주요 지역 | 사용자 활동 지역 |

가입 시 선택한 역할은 MVP에서 변경하지 않는다.

### 로그인/로그아웃

- 가입한 사용자는 이메일과 비밀번호로 로그인한다.
- 로그인한 사용자만 수요, 재고, 매칭, 문의 기능을 사용할 수 있다.
- 사용자는 로그아웃할 수 있다.

---

## 6. 권한 요구사항

| 기능 | 수요자 | 공급자 | 관리자 |
|------|--------|--------|--------|
| 수요 등록 | 가능 | 불가 | 불가 |
| 재고 등록 | 불가 | 가능 | 불가 |
| 본인 글 수정 | 가능 | 가능 | 불가 |
| 본인 글 삭제 | 가능 | 가능 | 불가 |
| 매칭 결과 조회 | 본인 수요 기준 | 본인 재고 기준 | 전체 조회 가능 |
| 문의 요청 | 가능 | 가능 | 불가 |
| 게시글 숨김 | 불가 | 불가 | 가능 |

관리자는 사용자의 글 내용을 직접 수정하거나 삭제하지 않는다.

숨김 처리된 게시글은 일반 사용자 목록과 매칭 대상에서 제외된다.

---

## 7. 수요 요구사항

### 수요 등록

수요자는 필요한 나무 정보를 등록한다.

필수 입력값:

| 항목 | 설명 |
|------|------|
| 수종 | 예: 왕벚나무, 소나무 |
| 수령/규격 | 예: 3년생, 10년생 |
| 수량 | 필요한 수량 |
| 필요 시기 | 납품 또는 사용이 필요한 시기 |
| 납품 주소 | 수요 위치 |
| 좌표 | 납품 주소를 변환한 위도/경도 |
| 예산 | 총 구매 가능 예산 |

### 수요 상태

| 상태 | 설명 |
|------|------|
| `open` | 매칭 가능한 공개 상태 |
| `matched` | 매칭 후보가 있는 상태 |
| `closed` | 사용자가 마감한 상태 |
| `hidden` | 관리자가 숨김 처리한 상태 |

### 수요 수정/삭제

- 수요자는 본인이 작성한 수요만 수정할 수 있다.
- 수요자는 본인이 작성한 수요만 삭제할 수 있다.
- 수요가 수정되면 매칭 후보를 다시 계산한다.
- 삭제된 수요는 목록과 매칭에서 제외된다.

---

## 8. 재고 요구사항

### 재고 등록

공급자는 판매 가능한 재고 정보를 등록한다.

필수 입력값:

| 항목 | 설명 |
|------|------|
| 수종 | 예: 왕벚나무, 소나무 |
| 수령/규격 | 예: 3년생, 10년생 |
| 수량 | 판매 가능 수량 |
| 단가 | 1그루당 가격 |
| 재고 주소 | 재고 위치 |
| 좌표 | 재고 주소를 변환한 위도/경도 |
| 사진 | 재고 사진 |
| 유효기간 | 재고가 유효한 기간 |

### 재고 상태

| 상태 | 설명 |
|------|------|
| `active` | 매칭 가능한 공개 상태 |
| `expired` | 유효기간이 지난 상태 |
| `hidden` | 관리자가 숨김 처리한 상태 |

### 재고 수정/삭제

- 공급자는 본인이 작성한 재고만 수정할 수 있다.
- 공급자는 본인이 작성한 재고만 삭제할 수 있다.
- 재고가 수정되면 매칭 후보를 다시 계산한다.
- 삭제된 재고는 목록과 매칭에서 제외된다.

---

## 9. 매칭 요구사항

### 매칭 생성 시점

매칭 후보는 아래 시점에 자동 생성 또는 재계산한다.

- 수요 등록
- 수요 수정
- 재고 등록
- 재고 수정
- 관리자의 게시글 숨김 처리

### 매칭 조건

수요와 재고는 아래 조건을 모두 만족해야 매칭 후보가 된다.

| 조건 | 기준 |
|------|------|
| 수종 | 수요 수종과 재고 수종이 일치 |
| 수령/규격 | 수요 수령/규격과 재고 수령/규격이 일치 |
| 수량 | 재고 수량이 수요 수량 이상 |
| 거리 | 수요 좌표와 재고 좌표 간 거리 50km 이내 |
| 가격 | `재고 단가 × 수요 수량 <= 수요 예산` |
| 상태 | 수요는 `open` 또는 `matched`, 재고는 `active` |

숨김 처리된 수요/재고는 매칭 대상에서 제외한다.

### 매칭 결과 조회

수요자는 본인 수요별로 매칭된 재고 후보를 조회한다.

공급자는 본인 재고별로 매칭된 수요 후보를 조회한다.

관리자는 전체 매칭 결과를 조회할 수 있다.

---

## 10. 문의 요청 요구사항

사용자는 매칭 후보에 대해 문의 요청을 남길 수 있다.

문의 요청은 채팅이 아니라 상태 데이터로 관리한다.

### 문의 상태

| 상태 | 설명 |
|------|------|
| `requested` | 문의 요청됨 |
| `accepted` | 상대가 문의를 수락함 |
| `rejected` | 상대가 문의를 거절함 |
| `cancelled` | 요청자가 문의를 취소함 |

### 문의 규칙

- 수요자는 본인 수요에 매칭된 재고 후보에 문의 요청을 보낼 수 있다.
- 공급자는 본인 재고에 매칭된 수요 후보에 문의 요청을 보낼 수 있다.
- 같은 매칭 후보에 중복 문의 요청을 만들지 않는다.
- 숨김 처리된 게시글에는 문의 요청을 보낼 수 없다.

---

## 11. 관리자 요구사항

관리자는 수요와 재고 게시글을 확인하고 문제 게시글을 숨김 처리할 수 있다.

숨김 처리 시 사유를 입력한다.

숨김 처리된 게시글은 아래 대상에서 제외된다.

- 일반 사용자 목록
- 수요/재고 상세 접근
- 매칭 후보 생성
- 문의 요청 생성

관리자는 MVP에서 사용자 계정 정지, 역할 변경, 게시글 직접 수정, 게시글 직접 삭제를 하지 않는다.

## 14. 검증 기준

- 수요자가 수요를 등록하면 조건에 맞는 재고 후보가 생성된다.
- 공급자가 재고를 등록하면 조건에 맞는 수요 후보가 생성된다.
- 수종이 다르면 매칭되지 않는다.
- 수령/규격이 다르면 매칭되지 않는다.
- 재고 수량이 수요 수량보다 적으면 매칭되지 않는다.
- 수요와 재고 간 거리가 50km를 초과하면 매칭되지 않는다.
- `재고 단가 × 수요 수량`이 수요 예산을 초과하면 매칭되지 않는다.
- 작성자가 아닌 사용자는 글을 수정/삭제할 수 없다.
- 관리자가 숨김 처리한 글은 목록과 매칭에서 제외된다.
- 매칭 후보에 대해 문의 요청을 남길 수 있다.
- 같은 매칭 후보에 중복 문의 요청이 생성되지 않는다.
