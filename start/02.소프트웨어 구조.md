# 02. 소프트웨어 구조

## 기본 방향

나무 장터는 프론트엔드와 백엔드를 분리해서 구성한다.

초기에는 웹 프로토타입으로 빠르게 검증하되, 백엔드 API와 도메인 로직을 독립적으로 설계해서 이후 모바일 앱, 관리자 화면, 매칭 고도화로 확장하기 쉽게 만든다.

```txt
사용자 브라우저
    ↓
Frontend: React + Vite + TypeScript
    ↓ REST API
Backend: Python FastAPI
    ↓
Database: Supabase PostgreSQL
```

---

## 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 프론트엔드 | React + Vite + TypeScript | 백엔드와 분리하기 쉽고, 구조가 단순하며, React 생태계를 활용할 수 있다. |
| 백엔드 | Python FastAPI | API 서버를 빠르게 만들 수 있고, 향후 매칭 알고리즘·검색·추천 로직을 Python으로 확장하기 좋다. |
| 데이터베이스 | Supabase PostgreSQL | PostgreSQL 기반으로 안정적이고, Supabase의 인증·스토리지·실시간 기능을 단계적으로 활용할 수 있다. |
| 파일 저장 | Supabase Storage | 재고 사진 업로드와 관리에 적합하다. |
| 지도 | Kakao Map API | 한국 주소, 지역, 좌표 처리에 적합하다. |
| 알림 | 초기에는 DB 기반 알림, 이후 Realtime/Push 확장 | MVP에서는 단순하게 시작하고, 필요 시 실시간 알림으로 확장한다. |

## 프론트엔드 구조

프론트엔드는 화면과 사용자 인터랙션을 담당한다.

API 호출은 백엔드로만 보내고, Supabase DB에 직접 접근하지 않는 것을 기본 원칙으로 한다. 이렇게 하면 권한, 검증, 매칭 로직을 백엔드에서 일관되게 관리할 수 있다.

```txt
frontend/src/
  app/
    router.tsx
    providers.tsx

  pages/
    DemandListPage.tsx
    DemandCreatePage.tsx
    StockListPage.tsx
    StockCreatePage.tsx
    MatchListPage.tsx

  features/
    demands/
      api.ts
      types.ts
      DemandForm.tsx
      DemandList.tsx

    stocks/
      api.ts
      types.ts
      StockForm.tsx
      StockList.tsx

    matches/
      api.ts
      types.ts
      MatchList.tsx

  shared/
    api/
      client.ts
    components/
    hooks/
    types/
```

### 프론트엔드 원칙

- 화면 단위 코드는 `pages/`에 둔다.
- 기능별 UI, 타입, API 호출은 `features/` 아래에 모은다.
- 공통 API 클라이언트, 버튼, 입력 컴포넌트 등은 `shared/`에 둔다.
- 프론트엔드는 백엔드 API 응답 타입을 기준으로 화면을 구성한다.
- 지도, 사진 업로드, 실시간 알림 같은 외부 연동은 기능별로 감싸서 사용한다.

## 백엔드 구조

백엔드는 인증, 권한, DB 접근, 입력 검증, 매칭 로직, 알림 생성을 담당한다.

```txt
backend/app/
  main.py

  core/
    config.py
    database.py
    security.py
    errors.py

  domains/
    users/
      router.py
      service.py
      schema.py
      repository.py

    demands/
      router.py
      service.py
      schema.py
      repository.py

    stocks/
      router.py
      service.py
      schema.py
      repository.py

    matches/
      router.py
      service.py
      schema.py
      repository.py

    notifications/
      router.py
      service.py
      schema.py
      repository.py

  integrations/
    supabase.py
    kakao_map.py
```

### 백엔드 계층 역할

| 파일 | 역할 |
|------|------|
| `router.py` | HTTP API 엔드포인트를 정의한다. |
| `schema.py` | 요청/응답 데이터 구조와 검증 규칙을 정의한다. |
| `service.py` | 업무 로직을 처리한다. 예: 수요 등록 후 매칭 후보 찾기 |
| `repository.py` | DB 조회와 저장을 담당한다. |
| `integrations/` | Supabase, Kakao Map 같은 외부 서비스 연동을 담당한다. |
| `core/` | 설정, DB 연결, 보안, 공통 예외 처리를 담당한다. |

## 주요 도메인

### users

사용자 정보를 관리한다.

- 구매자: 조경업체, 시공사 등 수요 등록자
- 공급자: 묘목장, 농장, 개인 판매자 등 재고 등록자

### demands

수요 게시글을 관리한다.

- 수종
- 수령 또는 규격
- 수량
- 필요 시기
- 납품 지역
- 예산
- 상태: `open`, `matched`, `closed`

### stocks

공급자의 재고를 관리한다.

- 수종
- 수령 또는 규격
- 수량
- 단가
- 위치
- 사진
- 유효기간

### matches

수요와 재고의 매칭 결과를 관리한다.

초기에는 수종, 수령, 수량, 지역을 기준으로 단순 매칭한다. 이후 거리, 가격, 신뢰도, 응답률 등을 반영한 점수 기반 매칭으로 확장한다.

### notifications

사용자에게 전달할 알림을 관리한다.

초기에는 DB에 알림 데이터를 저장하고 화면에서 조회하는 방식으로 시작한다. 이후 Supabase Realtime, 모바일 푸시 알림으로 확장한다.

## MVP 개발 순서

### 1단계: 프로젝트 뼈대

- `frontend/` React + Vite + TypeScript 초기화
- `backend/` FastAPI 초기화
- Supabase PostgreSQL 연결
- 기본 환경 변수 정리

### 2단계: 수요/재고 CRUD

- 수요 등록, 목록, 상세
- 재고 등록, 목록, 상세
- 입력값 검증
- 기본 사진 업로드

### 3단계: 단순 매칭

- 수종, 수령, 지역 기준 매칭
- 수요 등록 시 매칭 후보 생성
- 재고 등록 시 매칭 후보 생성
- 매칭 목록 화면 제공

### 4단계: 지도와 검색

- Kakao Map API 연동
- 주소를 좌표로 변환
- 지역/거리 기반 필터
- 수종, 수령, 수량 검색

### 5단계: 알림 확장

- DB 기반 알림 목록
- 읽음 처리
- Supabase Realtime 검토
- 모바일 앱 전환 시 푸시 알림 검토

## 구조 결정 요약

- 백엔드는 Python FastAPI로 구성한다.
- 프론트엔드는 React + Vite + TypeScript로 구성한다.
- Next.js는 초기 구조에서는 사용하지 않는다.
- 프론트엔드는 백엔드 API만 호출한다.
- DB 접근, 권한, 검증, 매칭 로직은 백엔드에 둔다.
- Supabase는 PostgreSQL, Storage, Realtime 기능을 단계적으로 사용한다.
- 모바일 앱이 필요해지면 React Native Expo를 별도 클라이언트로 추가한다.
