# 매칭 성능 최적화

## 문제 진단

매칭 탭을 열 때마다 PostGIS RPC → 전체 N건 enrichment → Python 정렬 → 30건 반환 순으로 처리되어 데이터가 많아질수록 느려지는 구조였다.

```
기존 흐름
──────────────────────────────────────────────
RPC → N건 전체 반환
  → enrich_pairs(N건) : demands 조회 + stocks 조회 (순차)
    → Python sort(N)
      → 30건만 응답
──────────────────────────────────────────────
```

핵심 문제 두 가지:

1. **전체 N건을 enrich한 뒤 30건만 쓴다** — 불필요한 DB 조회 N배
2. **demands 조회와 stocks 조회가 순차** — 직렬 네트워크 왕복 2회

---

## 구현된 최적화

### 1. 페이지네이션을 enrich 앞으로 이동

정렬·슬라이싱을 먼저 한 뒤 현재 페이지의 30건만 enrich한다.

```
개선된 흐름
──────────────────────────────────────────────
RPC → N건 반환 (UUID + 점수 필드만, 행 크기 작음)
  → Python 가중치 재계산 → 필터 → sort → 30건 슬라이스
    → enrich_pairs(30건) : demands + stocks 병렬 조회
      → 응답
──────────────────────────────────────────────
```

**효과**: enrich 대상이 N건 → 30건으로 고정. 데이터가 늘어도 enrich 비용은 일정.

관련 파일: [service.py](../../namu/backend/app/domains/matches/service.py)

### 2. enrich_pairs 병렬화

`ThreadPoolExecutor(2)`로 demands·stocks 조회를 동시에 실행한다.

```python
with ThreadPoolExecutor(max_workers=2) as ex:
    d_fut = ex.submit(fetch_demands)
    s_fut = ex.submit(fetch_stocks)
    demands = d_fut.result()
    stocks  = s_fut.result()
```

**효과**: 직렬 2회 → 병렬 1회. enrich 레이턴시 약 50% 감소.

### 3. SELECT 컬럼 최소화

`SELECT *` 대신 목록 카드에서 실제 사용하는 컬럼만 지정한다.

```python
_DEMAND_LIST_COLS = "id,species,spec,quantity,budget,user_id"
_STOCK_LIST_COLS  = "id,species,spec,quantity,unit_price,photo_urls,user_id"
```

상세 페이지(`/matches/:id`)는 기존처럼 `SELECT *` 유지.

### 4. match_cache 테이블 (로그인 시 pre-warm)

기본 가중치 + total_score 정렬 조합에 한해 DB 캐시를 사용한다.

| 항목 | 내용 |
|------|------|
| 캐시 키 | `(user_id, page, max_km)` |
| TTL | 2시간 (expires_at 컬럼) |
| 대상 | 기본 가중치 + sort_by = total_score 요청만 |
| 저장 위치 | `match_cache` 테이블 (Supabase) |
| 만료 정리 | `cleanup_expired_match_cache()` 함수 (pg_cron 또는 수동 호출) |

**pre-warm 동작**

1. 로그인 → 서버가 응답 반환 후 `BackgroundTasks`로 `prewarm_match_cache` 실행
2. 1~3페이지 90건을 한 번에 enrich → 캐시 적재
3. 이후 사용자가 매칭 탭을 열면 캐시 히트 → 즉시 반환

**캐시 무효화**

- 비기본 가중치로 요청이 오면 해당 유저 캐시 전체 삭제 후 실시간 조회
- 기본값으로 되돌리면 다음 요청에서 캐시 미스 → 실시간 조회 후 캐시 재적재
- TTL 2시간 초과 시 자동 만료

관련 마이그레이션: `017_match_cache.sql`

### 5. 프론트엔드 가중치 전달

기존에는 가중치가 localStorage에만 저장되고 API 요청에 포함되지 않았다.  
개선 후 `listMatches()`가 localStorage에서 가중치를 읽어 쿼리 파라미터로 전송한다.

```ts
// 매 요청마다 localStorage 가중치 자동 포함
listMatches(maxKm, page, sortBy, weights)
```

설정 페이지에서 저장 후 탭 포커스 복귀 시 `window.focus` 이벤트로 가중치를 재로드해 자동 반영한다.

## 향후 개선: Redis 캐시

!!! note "미구현 — 향후 도입 예정"
    현재는 Supabase `match_cache` 테이블을 사용한다.  
    트래픽이 늘어 DB 캐시 조회가 병목이 되면 Redis로 전환한다.

### 도입 배경

DB 캐시 방식의 한계:

- 캐시 조회도 Supabase 네트워크 왕복이 발생
- 동시 접속자가 늘면 캐시 테이블 자체가 부하를 받음
- TTL 기반 만료를 직접 관리해야 함

Redis는 인메모리 구조라 캐시 읽기 레이턴시가 1ms 이하이며 TTL 관리가 내장되어 있다.

### 예상 구조

```
캐시 키: match:{user_id}:{max_km}:{page}
값: JSON 직렬화된 {items, total}
TTL: 7200초 (2시간)
```

```python
# 예시 (미구현)
import redis
r = redis.Redis(host=REDIS_HOST, port=6379, decode_responses=True)

def get_cached_page(user_id, page, max_km):
    key = f"match:{user_id}:{max_km}:{page}"
    raw = r.get(key)
    return json.loads(raw) if raw else None

def set_cached_page(user_id, page, max_km, items, total):
    key = f"match:{user_id}:{max_km}:{page}"
    r.setex(key, 7200, json.dumps({"items": items, "total": total}))

def clear_user_cache(user_id):
    # SCAN으로 해당 유저 키 전체 삭제
    for key in r.scan_iter(f"match:{user_id}:*"):
        r.delete(key)
```

### 전환 조건

아래 중 하나라도 해당되면 Redis 도입을 검토한다.

- 동시 접속 사용자 50명 이상에서 매칭 응답 시간 1초 초과
- Supabase 쿼리 사용량에서 match_cache 테이블 조회 비중 10% 초과
- match_cache 테이블 행 수 10만 건 초과

### 인프라 옵션

| 옵션 | 특징 |
|------|------|
| Upstash Redis | 서버리스, 무료 구간 있음, Vercel/Render와 연동 쉬움 |
| Redis Cloud | 관리형, 30MB 무료 플랜 |
| Render Redis | 백엔드와 같은 플랫폼, 네트워크 레이턴시 최소 |

현재 백엔드가 Render에 배포되어 있으므로 **Render Redis**가 가장 간단한 선택.

### 마이그레이션 계획

1. `requirements.txt`에 `redis` 추가
2. `repository.py`의 캐시 CRUD 함수를 Redis 구현으로 교체 (인터페이스 동일)
3. `match_cache` 테이블 DROP
4. 환경 변수 `REDIS_URL` 추가

인터페이스(`get_cached_page`, `set_cached_page`, `clear_user_cache`)가 그대로이므로 `service.py`와 `users/router.py`는 수정 없이 전환 가능.
