# 거래 신뢰도 시스템

## 설계 원칙

별점을 처음부터 크게 반영하면 데이터가 적을 때 왜곡이 생긴다.  
예를 들어 판매자 A가 평가 1개에 5점, 판매자 B가 평가 30개에 4.6점이면 실제 신뢰도는 B가 더 높다.  
**작게 시작하고 데이터가 쌓이면 비중을 키우는 방식**으로 진행한다.

---

## 평가 구조

### 공통 규칙

- 거래 완료(`transactions.status = 'settled'`) 후에만 평가 가능
- 구매자 → 판매자 평가 / 판매자 → 구매자 평가, 양방향 모두 수집
- 거래 1건당 1회 (`UNIQUE(transaction_id, reviewer_id)`)

### 판매자 평가 태그 (구매자가 작성)

`응답 빠름` / `사진과 일치` / `거래 원활` / `정보 정확` / `약속 이행`

### 구매자 평가 태그 (판매자가 작성)

`문의 구체적` / `약속 이행` / `의사결정 명확` / `무리한 요구 없음`

### 입력 형식

```
별점: 1–5점 + 태그 복수 선택
```

## 신뢰도 점수 계산

평균 별점을 그대로 쓰지 않고 **신뢰도 점수(0–100)**로 변환한다.

### 공식

```
신뢰도 점수 =
  Bayesian 별점 / 5 × 100 × 0.7    (70%)
  + 거래완료율 × 100 × 0.2          (20%)
  + 응답률 × 100 × 0.1              (10%)
```

### Bayesian 평균

```
Bayesian 평균 = (C × M + n × avg_rating) / (C + n)

C = 10     (기준 거래 수)
M = 3.5    (전체 사용자 평균 별점)
n = 실제 평가 수
```

평가 0개 → 전체 평균 3.5점 수렴 → **신규 사용자 자동 70점 시작**.

### 데이터 소스

| 지표 | 소스 |
|------|------|
| 평균 별점 | `reviews` 테이블 |
| 거래 완료율 | `transactions.status = 'settled'` 비율 |
| 응답률 | `inquiries.receiver_id` 기준 `accepted/rejected` 비율 |

### trust_score 자동 갱신

`reviews` INSERT/UPDATE 시 PostgreSQL 트리거가 `compute_trust_score(reviewee_id)` 를 호출하고  
`users.trust_score` 를 즉시 업데이트한다. 별도 배치 없이 실시간 반영.

```sql
-- 트리거 (018_reviews.sql)
CREATE TRIGGER reviews_refresh_trust
    AFTER INSERT OR UPDATE ON reviews
    FOR EACH ROW EXECUTE FUNCTION refresh_trust_score();
```

## 매칭 점수 반영

### 핵심 설계: 같은 그룹 ×0은 RPC 구조로 자동 보장

구매자용 RPC는 **판매자** 신뢰도만, 판매자용 RPC는 **구매자** 신뢰도만 조인한다.  
별도로 ×0 처리를 코딩할 필요 없이 구조적으로 분리된다.

```
find_matches_for_buyer  →  stocks.user_id → users.trust_score  (판매자 신뢰도)
find_matches_for_seller →  demands.user_id → users.trust_score (구매자 신뢰도)
find_all_matches        →  stocks.user_id → users.trust_score  (판매자 신뢰도 기준)
```

### 가중치

```sql
-- 019_matching_trust.sql
total_score = dist_score * 100
            + spec_score * 10
            + sz_score   * 1
            + pr_score   * 5
            + trust_score * 2   -- 초기 ~10% 비중 (고정, 사용자 조정 불가)
```

신규 사용자 trust_score = 70이므로 기본 기여값 = 140점 (전체 평균과 동일 → 불이익 없음).

### 단계별 로드맵

| 단계 | 내용 | 상태 |
|------|------|------|
| 1단계 | reviews 테이블 + 평가 API + 매칭 카드 신뢰도 표시 | **완료** |
| 2단계 | 매칭 RPC에 trust_score × 2 반영 | **완료** |
| 3단계 | 데이터 검증 후 가중치 확대 (trust_score × N) | 미정 |

| 엔드포인트 | 설명 |
|-----------|------|
| `POST /api/reviews` | 평가 생성 (settled 거래만 허용, 거래당 1회) |

### 프론트엔드

- **`InquiryDetailPage`**: `quote.status === 'settled'` 시 평가 섹션 자동 노출
  - 별점(★ 1–5) + 역할별 태그 멀티 선택
  - 제출 후 "평가 완료" 상태로 전환
- **`MatchListPage` 카드**: trust_score가 있으면 5번째 점수 바(`신뢰`) 표시
  - 기존 4칸 그리드 → `grid-cols-5` 자동 전환

## 구매자 평가의 활용 방향

구매자 평가는 **매칭 알고리즘**보다 **문의·거래 화면의 신뢰 표시**에 더 적합하다.

- 판매자가 문의 목록을 볼 때 구매자 신뢰도 확인 수단
- 거래 수락 여부 판단 보조
- 매칭 점수보다는 "이 사람과 거래해도 괜찮은가"의 판단 지표

단, 현재 구현에서는 `find_matches_for_seller` RPC가 구매자 trust_score를 조인하므로  
판매자 입장의 매칭 순위에도 구매자 신뢰도가 반영된다 (×2 고정).
