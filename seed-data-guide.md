# 샘플 데이터 가이드

## 목적

나무 장터의 데모/테스트 화면을 실제 서비스처럼 보이게 하기 위해 수요, 재고, 매칭, 문의 데이터를 충분히 생성한다. 현재 목표는 기존 약 200건 수준의 수요/재고를 약 5배인 1,000건 수준으로 늘리는 것이다.

## 현재 목표 수량

| 항목 | 목표 |
|---|---:|
| users | 샘플 계정 30명 이상 |
| demands | 1,000건 |
| stocks | 1,000건 |
| matches | 조건에 따라 자동 계산 |
| inquiries | 500건 |
| 수요 사진 | row당 3장 |
| 재고 사진 | row당 3장 |

## 실행 스크립트

대량 샘플 데이터는 다음 스크립트가 담당한다.

```powershell
.\backend\.venv\Scripts\python tests\seed_scaleup_5x.py
```

스크립트 특징:

- 기존 데이터를 삭제하지 않는다.
- 현재 개수를 확인한 뒤 목표 수량까지 부족한 만큼만 추가한다.
- `assets/seed/stock-images/*.webp` 이미지를 Supabase Storage의 `stock-photos/catalog/` 경로에 한 번만 업로드한다.
- 각 수요/재고 row에는 catalog 이미지 URL 3개를 순환 배정한다.
- 이미지 파일을 row마다 중복 업로드하지 않으므로 Storage 사용량 증가를 줄인다.
- 수요/재고 삽입 후 매칭을 다시 계산한다.
- 문의가 500건 미만이면 매칭 후보를 기준으로 문의 row를 추가한다.

## 이미지 구성

로컬 이미지 경로:

```text
assets/seed/stock-images/
```

현재 이미지는 `stock-tree-001.webp` 형식으로 관리한다. 새 이미지가 필요하면 AI 생성 contact sheet를 만든 뒤 4x4 타일로 잘라 다음 번호부터 저장한다.

이미지 업로드 경로:

```text
stock-photos/catalog/stock-tree-001.webp
```

row별 경로에 같은 이미지를 반복 업로드하지 않고 catalog URL을 공유한다. 이렇게 하면 1,000개 재고와 1,000개 수요가 각각 3장씩 이미지를 갖더라도 실제 Storage 파일 수는 이미지 풀 크기만큼만 증가한다.

## 규격 표기

현재 샘플 데이터는 조경수 거래에서 흔히 쓰는 규격 문자열을 그대로 저장한다.

| 표기 | 의미 |
|---|---|
| H | 수고 |
| W | 수관폭 |
| R | 근원직경 |
| B | 흉고직경 |

예시:

```text
H3.0xR8
H4.0xB12
H2.0xW1.2
H0.8xW0.6
```

교목은 `H`, `R`, `B`를 중심으로 하고, 관목과 상록 수형은 `H`, `W`를 많이 사용한다.

## 가격/수량 범위

샘플 가격은 실제 견적 기준이 아니라 화면 테스트용 범위다.

| 구분 | 수량 예시 | 단가 범위 |
|---|---:|---:|
| 교목 | 20~200주 | 45,000~190,000원 |
| 상록/침엽 | 20~200주 | 30,000~260,000원 |
| 관목 | 20~1,000주 | 3,000~30,000원 |

수요 예산은 `수량 * 예상 단가`로 만들고, 재고 총액이 수요 예산 이하일 때 매칭될 수 있도록 일부 여유율을 둔다.

## 매칭 생성 기준

백엔드 매칭 조건은 다음과 같다.

1. 수요와 재고의 `species`가 같다.
2. 수요와 재고의 `spec`이 같다.
3. 재고 수량이 수요 수량 이상이다.
4. `재고 단가 * 수요 수량`이 수요 예산 이하이다.
5. 수요 좌표와 재고 좌표의 직선거리가 300km 이하이다.

거리는 `backend/app/domains/matches/service.py`의 haversine 계산과 같은 방식으로 계산한다. 화면의 50km, 100km, 200km 필터는 저장된 `distance_km`에 대한 추가 필터다.

## Supabase 용량 관리

현재 데이터는 row 수 자체보다 이미지 Storage 사용량이 더 중요하다.

확인 위치:

```text
Supabase Dashboard > Organization > Usage
```

주요 확인 항목:

- Database Size
- Storage Size
- Egress

Free 기준에서 Database Size는 500MB, Storage는 1GB를 특히 확인한다. 현재처럼 catalog 이미지 URL을 공유하면 row를 늘려도 Storage 증가가 완만하다.

## 재실행 주의사항

- `tests/seed_scaleup_5x.py`는 목표 수량까지만 추가하므로 같은 목표로 다시 실행하면 대부분 추가 삽입하지 않는다.
- 목표 수량을 늘리고 싶으면 스크립트 상단의 `TARGET_DEMANDS`, `TARGET_STOCKS`, `TARGET_INQUIRIES`를 수정한다.
- 이미지 수를 늘리고 싶으면 `assets/seed/stock-images`에 `stock-tree-XXX.webp` 파일을 추가한 뒤 스크립트를 다시 실행한다.
- 기존 row의 사진 URL을 강제로 다시 배정하는 스크립트는 아니다.
